## Template Contribution Standards

### Template Quality Guidelines

All community templates must meet these quality standards.

<AccordionGroup>
  <Accordion title="Template Structure Standards">
    **Required Files**:
    - `template.yml` with complete metadata
    - `README.md` with usage instructions
    - `CHANGELOG.md` for version history
    - `LICENSE` file with appropriate license
    
    **Recommended Files**:
    - `examples/` directory with usage examples
    - `tests/` directory with template tests

    - `docs/` for detailed documentation
  </Accordion>

  <Accordion title="Code Quality Requirements">
    **Template Code**:
    - Follow language-specific style guides
    - Include comprehensive error handling
    - Provide meaningful variable names
    - Add inline documentation for complex logic
    
    **Generated Code**:
    - Must be production-ready
    - Follow security best practices
    - Include appropriate tests
    - Meet accessibility standards (for UI components)
  </Accordion>

  <Accordion title="Documentation Standards">
    **Required Documentation**:
    - Clear description of template purpose
    - Complete usage instructions with examples
    - Variable documentation with types and validation
    - Generated file structure explanation
    
    **Best Practices**:
    - Include screenshots/demos for visual components
    - Provide troubleshooting section
    - Add related templates and resources

### Template Testing Framework

All community templates must include comprehensive tests.

```typescript
// tests/template.test.ts - Template testing example
import { TemplateTestSuite, HypergenTestEngine } from '@hyperdev/testing';

describe('React TypeScript Component Template', () => {
  let testEngine: HypergenTestEngine;
  
  beforeAll(() => {
    testEngine = new HypergenTestEngine('./template.yml');
  });

  describe('Template Validation', () => {
    test('should have valid template metadata', async () => {
      const validation = await testEngine.validateTemplate();
      expect(validation.valid).toBe(true);
      expect(validation.errors).toHaveLength(0);
    });

    test('should validate all required variables', async () => {
      const variables = await testEngine.getRequiredVariables();
      expect(variables).toContain('name');
      expect(variables).toContain('typescript');
    });
  });

  describe('Code Generation', () => {
    test('should generate valid TypeScript component', async () => {
      const result = await testEngine.generate({
        name: 'TestComponent',
        typescript: true,
        includeTests: true
      });

      expect(result.files).toContainFile('TestComponent.tsx');
      expect(result.files).toContainFile('TestComponent.test.tsx');
      
      const componentCode = result.files.find(f => 
        f.path.endsWith('TestComponent.tsx')
      )?.content;
      
      expect(componentCode).toContain('interface TestComponentProps');
      expect(componentCode).toMatch(/const TestComponent: React\.FC/);
    });

    test('should generate compilable TypeScript', async () => {
      const result = await testEngine.generate({
        name: 'CompileTest',
        typescript: true
      });

      const tsCompilation = await testEngine.compileTypeScript(result.files);
      expect(tsCompilation.errors).toHaveLength(0);
    });
  });

  describe('Output Quality', () => {
    test('should generate code that passes ESLint', async () => {
      const result = await testEngine.generate({
        name: 'LintTest',
        typescript: true
      });

      const lintResults = await testEngine.runESLint(result.files);
      expect(lintResults.errorCount).toBe(0);
    });

    test('should generate accessible components', async () => {
      const result = await testEngine.generate({
        name: 'AccessibilityTest',
        includeAccessibility: true
      });

      const a11yResults = await testEngine.runAccessibilityCheck(result.files);
      expect(a11yResults.violations).toHaveLength(0);
    });
  });
});
```

### Code Quality Standards

All code contributions must meet these quality standards.

<CodeGroup>
```typescript TypeScript Standards
// Use strict TypeScript configuration
export interface ComponentConfig {
  readonly name: string;
  readonly type: 'functional' | 'class';
  readonly props?: Record<string, unknown>;
}

// Prefer composition over inheritance
export class TemplateGenerator {
  constructor(
    private readonly config: ComponentConfig,
    private readonly validator: InputValidator,
    private readonly renderer: TemplateRenderer
  ) {}

  async generate(): Promise<GenerationResult> {
    // Always validate inputs
    const validationResult = await this.validator.validate(this.config);
    if (!validationResult.isValid) {
      throw new ValidationError(validationResult.errors);
    }

    return this.renderer.render(this.config);
  }
}
```

```javascript Testing Standards
// Comprehensive test coverage
describe('TemplateGenerator', () => {
  let generator: TemplateGenerator;
  
  beforeEach(() => {
    generator = new TemplateGenerator(
      mockConfig,
      mockValidator,
      mockRenderer
    );
  });

  describe('input validation', () => {
    test('should reject invalid component names', async () => {
      await expect(generator.generate({
        name: '123invalid'
      })).rejects.toThrow(ValidationError);
    });
  });

  describe('error handling', () => {
    test('should handle template rendering errors gracefully', async () => {
      mockRenderer.render.mockRejectedValue(new Error('Template error'));
      
      await expect(generator.generate()).rejects.toThrow('Template error');
    });
  });
});
```

### Pull Request Process

Follow this process for code contributions.

<Steps>
  <Step title="Preparation">
    - Create feature branch from `main`
    - Write comprehensive tests for new functionality
    - Update documentation for any API changes
    - Run full test suite locally
  </Step>

  <Step title="Pull Request Creation">
    - Use descriptive PR title following conventional commits
    - Complete PR template with detailed description
    - Link related issues and discussions
    - Request appropriate reviewers
  </Step>

  <Step title="Review Process">
    - Address reviewer feedback promptly
    - Maintain clean commit history
    - Keep PR focused and atomic
    - Ensure CI/CD passes all checks
  </Step>

  <Step title="Merge Requirements">
    - Minimum 2 approvals from core contributors
    - All CI/CD checks must pass
    - Documentation must be updated
    - Breaking changes require RFC discussion

## Code of Conduct

### Community Standards

We are committed to providing a welcoming and inclusive experience for everyone.

**Our Standards**:
- Use welcoming and inclusive language
- Respect differing viewpoints and experiences
- Accept constructive criticism gracefully
- Focus on what is best for the community
- Show empathy towards other community members

**Unacceptable Behavior**:
- Harassment, discrimination, or intimidation
- Trolling, insulting, or derogatory comments
- Public or private harassment
- Publishing others' private information
- Other conduct inappropriate in a professional setting

### Enforcement

Community guidelines are enforced by our moderation team.

**Reporting Issues**:
- Email: community@hyperdev.org
- Discord: Direct message to moderators
- GitHub: Use issue templates for code-related concerns

**Consequences**:
- Warning for minor violations
- Temporary suspension for repeated violations
- Permanent ban for severe violations or repeated offenses
